Super Search

Custom Service Portal search experience for ServiceNow. The app provides a single search entry point that aggregates knowledge articles, catalog items and record producers, news, users, and topics into one ranked results experience.

What This App Does

Super Search replaces a single-source search box with a custom Service Portal search flow that queries multiple ServiceNow tables at request time, scores the candidate records in memory, and renders a normalized set of result cards in one consistent UI.

The app is widget-driven and does not depend on AI Search or a dedicated external indexing pipeline. Instead, it uses a scoped script include to query source tables directly and shape results for the portal layer.

Supported result types: Knowledge articles, catalog items, record producers, news articles, users, and topics.

Goals

  • Provide one search entry point from the portal front page.
  • Search across multiple ServiceNow content types with one result-card model.
  • Keep portal visibility aligned with portal taxonomy and Service Portal visibility rules.
  • Support type-based filtering on the results page.
  • Publish search and click analytics for reporting.

Main Components

1. superSearchEngine script include

This is the core search engine and the most important technical asset in the app.

  • Normalizes and tokenizes user queries
  • Expands queries using synonym sets
  • Builds per-table search passes
  • Queries multiple tables for candidates
  • Applies source-specific visibility and portal rules
  • Scores and sorts candidates in memory
  • Shapes results into a UI-friendly DTO

Public entry point: searchKnowledge(options)

2. Homepage widget

  • Renders the front-page search input
  • Supports compact mode for header/navigation use
  • Redirects to the results page with the user query in the URL
  • Does not perform the actual search itself

3. Results widget

  • Invokes the script include to execute the search
  • Renders sidebar filters, result cards, and pagination
  • Performs client-side filtering and paging on allResults
  • Publishes search impression and click analytics

4. Analytics bridge

  • Transforms widget analytics payloads into GlideSPSearchAnalytics payloads
  • Publishes search events and click events
  • Backfills click rank to sys_search_event when needed

Important: this dependency lives outside the scoped app export tree and must exist in the target instance.

End-to-End Flow

Homepage flow

  1. User enters a query in the homepage search widget.
  2. The client controller normalizes whitespace.
  3. The widget redirects to the configured results page.
?id=search_results&q=<query>

Results page flow

  1. Searchresults server script reads URL params and widget options.
  2. On initial page load without widget input, it builds a shell state and sets deferInitialQuery = true.
  3. The client controller immediately calls the server with the query.
  4. The server script calls superSearchEngine.searchKnowledge(...).
  5. The engine returns normalized results, filter counts, and paging metadata.
  6. The client handles filter changes and pagination locally from allResults.
  7. Search and click analytics are published by the results widget.

Search Strategy

Query normalization

  • Trims leading and trailing whitespace
  • Collapses repeated whitespace
  • Lowercases normalized values for matching and scoring
  • Tokenizes the query for overlap scoring and fallback logic

Synonym expansion

  • Synonyms come from ts_synonym_set.
  • If synonym_dictionary_id is configured, only that set is used.
  • Otherwise all active synonym sets are eligible.
  • Synonym-derived matches are scored lower than direct query matches.

Per-table pass definitions

The engine builds ordered search passes instead of issuing one broad query. Examples:

Knowledge
number, short_description, meta, keywords, optionally text
Catalog
name, short_description, meta, optionally description
News
title
Users
name, title, department.name, user_name, email
Topics
name, optionally description

Short-query optimization

Very short queries can produce too many broad matches. The app supports a short-query profile controlled by:

  • short_query_length
  • short_query_candidate_limit
  • short_query_result_limit

When the query length is at or below the threshold, body search is disabled and both candidate fetch count and exposed result count are reduced.

Base Visibility Filters

Knowledge
  • active = true
  • workflow_state = published
  • published <= today
  • valid_to empty or in the future
Catalog
  • active = true
  • hidden_sp = false
  • visible_sp = true
  • current user can view item in catalog search
News
  • active = true
  • content_type = configured news content type
Users
  • active = true
Topics
  • active = true
  • taxonomy belongs to current portal
  • optional featured topic subtree exclusion
Design choice: catalog items and topics are queried with GlideRecord rather than GlideRecordSecure so visibility follows portal configuration rules instead of raw table ACL behavior. Catalog and record producer user criteria are enforced with the Service Catalog API before results are returned.

Portal-Aware Behavior

Portal taxonomy drives topic visibility

The engine reads m2m_sp_portal_taxonomy for the active portal and only returns topics from those taxonomies.

Catalog items must be connected to visible topics

Catalog candidates are post-filtered through m2m_connected_content. Only items connected to topics in the current portal taxonomy are kept, which prevents the widget from surfacing catalog items that are not actually wired into the portal’s navigational model.

Catalog user criteria is enforced server-side

Catalog candidates are checked with sn_sc.CatItem(...).canViewOnSearch(false) before they are returned to the result widget. Inaccessible catalog items and record producers are kept out of result cards, paging, filters, and analytics payloads.

Ranking and Sorting

After collection, candidates from all sources are merged and scored in memory. The score is based on title, metadata, body text, and synonym matches.

  • Title exact, prefix, and contains matches have the strongest weight.
  • Metadata is secondary.
  • Body text has the lowest weight.
  • Synonym matches score below direct query matches.
  • Fallback news matches receive a score penalty.

Result type priority

Sorting is not score-only. The engine also applies a fixed type priority:

  1. catalog_item
  2. knowledge
  3. news
  4. topic
  5. sys_user
Implication: a lower-scoring catalog item can still rank above a higher-scoring knowledge article because result type priority is evaluated before score.

Result Model Returned to the Widget

The engine shapes source records into a normalized result object to keep the Angular template simple and source-agnostic.

{ sysId, resultType, resultTypeLabel, title, titleHtml, snippet, snippetHtml, number, kbName, catalogName, employeeTitle, departmentName, categoryName, language, updatedOnDisplay, url, isRequestItem, isNewsItem, isUser, isTopic }

Filtering Model

The results widget supports these filter ids:

all knowledge_total featured_kb knowledge_articles news catalog_item sys_user topic
  • knowledge_total is the superset of all knowledge hits.
  • featured_kb is a subset of knowledge based on a configured knowledge base sys_id.
  • knowledge_articles is knowledge excluding the featured KB subset.
  • Filter changes and paging are client-side operations against allResults.

Localization

  • UI labels in templates and defaults are mostly hardcoded in Norwegian.
  • Search execution itself is not fully language-aware across all sources.
  • Catalog and record producer display values now prefer display values for translated fields.

For catalog results, the engine now prefers getDisplayValue(field) for:

  • name
  • description
  • short_description
  • meta
Important distinction: localized display on the result card and language-aware search matching are different concerns. The former is implemented; the latter depends on how source fields are queried and translated.

Analytics

Search impression event

  • Normalized query
  • Portal id
  • Page id
  • Page size
  • Whether results were returned
  • Top results with sys_id and result type

Click event

  • Query
  • Portal id
  • Page id
  • Click rank
  • Browser info
  • Clicked record sys_id and result type
  • Visible result list used as ranking context

The global bridge maps result types to source tables and publishes them through GlideSPSearchAnalytics. It also updates the latest matching sys_search_event to keep click_rank usable for reporting.

Known Design Decisions and Limitations

  • Result type priority is opinionated and gives catalog items top priority.
  • Most user-facing labels are currently Norwegian and not externalized.
  • The solution performs live table queries at runtime rather than using a dedicated search index.
  • The initial results-page search is deferred to the client after the shell state is rendered.
  • The analytics bridge is a global dependency outside the scoped app export tree.
  • Localized display of catalog content is implemented, but language-aware search matching remains a separate problem.

Extension Guidance

To add another result source

  1. Add a table constant and context field map.
  2. Add candidate collection logic.
  3. Add base filters.
  4. Add a _store...Candidate method.
  5. Add scoring metadata and icon handling.
  6. Update analytics result-type-to-table mapping in the global bridge.

To change ranking

Start with _calculateScore, _getResultTypePriority, and _compareCandidates.

To change portal visibility rules

Focus on _applyCatalogBaseFilters, _applyTopicBaseFilters, _getConnectedCatalogItemIdMap, and _getPortalTaxonomyIds.

To change displayed fields

Modify the relevant _store...Candidate method, then update _shapeResults and the results widget template.
